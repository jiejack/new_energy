# Andrej Karpathy LLM-wiki 概念

## 概述

Andrej Karpathy 在 2026 年 4 月提出了 LLM-wiki 的概念，这是一种将知识从传统 RAG（检索增强生成）的解释器模式升级为编译器模式的架构。

## 核心思想

### 与传统 RAG 的区别

| 维度 | 传统 RAG | LLM Wiki |
|------|---------|----------|
| 知识处理 | 查询时检索 | 录入时编译 |
| 积累性 | 无，每次从零开始 | 持续增长 |
| 交叉引用 | 无 | 自动维护 |
| 矛盾检测 | 无 | 主动标记 |

### 核心理念

LLM-wiki 让 AI 在录入时就把知识编译成结构化的维基页面，而不是每次查询时从文档重新拼凑答案。知识持续积累、自动交叉引用、主动检测矛盾。

## 三层架构

### 1. Raw Sources（原始来源）
- **特性**：不可变的源文档
- **内容**：文章、论文、图片、数据文件
- **操作**：LLM 只读不写
- **存放**：`raw/` 目录

### 2. The Wiki（维基）
- **特性**：LLM 生成和维护的 Markdown 页面
- **内容**：
  - 摘要页
  - 实体页（人物、公司、产品...）
  - 概念页（技术、理论、方法...）
  - 对比分析
  - 综合综述
- **存放**：`wiki/` 目录

### 3. The Schema（模式）
- **特性**：配置文件
- **内容**：如 AGENTS.md 或 CLAUDE.md
- **作用**：定义 LLM 的行为规范和工作流

## 核心操作

### 1. Ingest（录入）
当用户提供新来源时，LLM 自动执行：
1. 读取原始来源，与用户讨论关键要点
2. 创建来源摘要页（`wiki/sources/`）
3. 识别实体和概念，创建或更新对应页面
4. 建立双向链接，页面间交叉引用
5. 更新分类目录（`wiki/index.md`）
6. 追加操作日志（`wiki/log.md`）

### 2. Query（查询）
回答问题时：
1. 先读 `wiki/index.md` 找到相关页面
2. 读取相关页面内容
3. 综合回答，引用来源
4. 如果答案有独立价值，建议用户将其存为新的 wiki 页面

### 3. Lint（校验）
定期健康检查：
- 页面间的事实矛盾
- 被提及但不存在的链接
- 没有入站链接的孤立页面
- 可以合并的重复内容
- 缺失 frontmatter 的页面
- 值得深入调研的知识缺口

## 关键文件

### index.md
按分类组织，每个页面一行：
```markdown
## 实体
- [页面标题](entities/xxx.md) — 一句话摘要
## 概念
- [页面标题](concepts/xxx.md) — 一句话摘要
## 来源
- [页面标题](sources/xxx.md) — 一句话摘要
```

### log.md
每条记录格式：
```markdown
## [YYYY-MM-DD] 操作类型 | 标题
简要描述做了什么、更新了哪些页面。
```

操作类型：ingest / query / lint / update

## 核心优势

1. **持续积累**：知识被编译一次后持续更新，而非每次重新推导
2. **自动交叉引用**：LLM 自动维护页面间的链接
3. **矛盾检测**：主动标记不同来源的冲突
4. **复利效应**：摄入的内容越多，新材料被解读时所处的上下文就越丰富

## 与本项目的关系

本项目采用 LLM-wiki 架构思想，将知识库存放于 `/doc/wiki/` 目录，AGENT.md 作为 schema 文件，实现知识的持续积累和管理。
