## 1. 系统概述

### 1.1 项目背景

新能源在线监控系统是面向光伏、风电、储能等新能源电站的综合监控平台，实现对设备运行状态的实时采集、处理、告警与分析。系统需支持百万级采集点位、千万级历史数据存储，并融合AI技术实现智能化运维。

### 1.2 系统目标

- **高性能**：支持100万采集点位并发采集，日增1000万历史数据
- **高可靠**：系统可用性≥99.9%，数据不丢失
- **高扩展**：支持水平扩展，适应业务增长
- **智能化**：融合AI技术，实现智能问答、智能配置、智能操作

### 1.3 性能指标

| 指标 | 目标值 |
|------|--------|
| 采集点位数 | 100万 |
| 定时任务数 | 20万 |
| 日数据增量 | 1000万条 |
| 年数据存储量 | ≥36亿条 |
| 查询响应时间 | 平均≤200ms，95%≤500ms |
| 任务调度延迟 | ≤1秒 |
| 采集延迟 | ≤100ms |
| 采集误差 | ≤0.1% |

## 2. 架构设计原则

### 2.1 领域驱动设计（DDD）

采用领域驱动设计方法，将业务领域划分为多个限界上下文，每个上下文对应一个微服务：

- **配置上下文**：区域、厂站、设备、采集点配置管理
- **采集上下文**：多协议数据采集与处理
- **告警上下文**：实时告警检测与通知
- **计算上下文**：逻辑计算与规则引擎
- **统计上下文**：多维度统计分析
- **存储上下文**：数据存储与查询优化
- **AI上下文**：智能问答与辅助决策

### 2.2 微服务架构

采用微服务架构，实现服务间低耦合、高内聚：

- 服务独立部署、独立扩展
- 服务间通过API或消息队列通信
- 每个服务拥有独立的数据存储

### 2.3 云原生设计

- 容器化部署（Docker）
- 容器编排（Kubernetes）
- 服务网格（可选）
- 声明式配置

## 4. 服务设计

### 4.1 服务清单

| 服务名 | 功能描述 | 技术栈 | 实例数 |
|--------|----------|--------|--------|
| api-server | RESTful API服务，提供配置管理、数据查询接口 | Go + Gin | 2-10 |
| collector | 数据采集服务，支持多协议采集 | Go + 协程池 | 3-20 |
| alarm | 告警服务，实时检测与通知 | Go + 规则引擎 | 2-5 |
| compute | 计算服务，公式计算与规则执行 | Go + 表达式引擎 | 2-5 |
| ai-service | AI服务，智能问答与辅助决策 | Go + LLM | 2-5 |
| scheduler | 调度服务，定时任务管理 | Go + Cron | 1-3 |

### 4.2 服务通信

#### 4.2.1 同步通信

```
┌─────────────┐      gRPC       ┌─────────────┐
│  api-server │ ◄──────────────► │   alarm     │
└─────────────┘                  └─────────────┘
       │
       │ gRPC
       ▼
┌─────────────┐      gRPC       ┌─────────────┐
│   compute   │ ◄──────────────► │ ai-service  │
└─────────────┘                  └─────────────┘
```

#### 4.2.2 异步通信

```
┌─────────────┐                  ┌─────────────┐
│  collector  │ ──── Kafka ────► │    alarm    │
└─────────────┘   (采集数据)      └─────────────┘
       │
       │ Kafka (设备状态)
       ▼
┌─────────────┐                  ┌─────────────┐
│   compute   │ ──── Kafka ────► │  storage    │
└─────────────┘   (计算结果)      └─────────────┘
```

### 4.3 消息队列主题设计

| 主题 | 生产者 | 消费者 | 说明 |
|------|--------|--------|------|
| nem.data.collect | collector | alarm, compute | 采集数据 |
| nem.alarm.event | alarm | notify | 告警事件 |
| nem.alarm.notify | alarm | sms, email, push | 告警通知 |
| nem.device.status | collector | api-server | 设备状态 |
| nem.compute.result | compute | storage | 计算结果 |

## 5. 数据架构

### 5.1 数据存储策略

| 数据类型 | 存储系统 | 保留策略 | 说明 |
|----------|----------|----------|------|
| 配置数据 | PostgreSQL | 永久 | 区域、设备、采集点配置 |
| 实时数据 | Redis | 1天 | 最新采集值、设备状态 |
| 历史数据 | Doris/ClickHouse | 1年+ | 时序数据，支持压缩 |
| 告警数据 | PostgreSQL + Redis | 1年 | 实时告警Redis，历史告警PG |
| 统计数据 | Doris/ClickHouse | 3年 | 统计报表数据 |

### 5.2 数据分片策略

#### 5.2.1 时序数据分片

```
按时间分区：按天分区，便于数据清理和查询
按厂站分片：同一厂站数据存储在同一分片，提高查询效率
```

#### 5.2.2 配置数据分库

```
按区域分库：大型区域独立数据库
读写分离：主库写入，从库读取
```

### 5.3 数据流设计

```

│  (采集服务)  │      │  (消息队列)  │

       │                   │
```

## 7. 安全设计

### 7.1 认证授权

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│   用户      │ ───► │ API Gateway │ ───► │   服务      │
└─────────────┘      └──────┬──────┘      └─────────────┘
                            │
                            │ JWT Token验证
                            ▼
                     ┌─────────────┐
                     │    Auth     │
                     │   Service   │
                     └─────────────┘
```

### 7.2 数据安全

- **传输加密**：HTTPS/TLS
- **存储加密**：敏感数据加密存储
- **访问控制**：RBAC权限模型
- **审计日志**：操作日志记录

### 7.3 网络安全

- 网络隔离：服务间通过Service Mesh通信
- 防火墙：限制端口访问
- DDoS防护：限流、熔断

## 9. 扩展性设计

### 9.1 水平扩展

- 无状态服务：通过增加实例扩展
- 有状态服务：通过分片扩展

### 9.2 垂直扩展

- 数据库：增加CPU、内存
- 缓存：增加内存

### 9.3 协议扩展
