## 1. 快速开始

### 1.1 安装 Git Hooks

```bash

cd e:\ai_work\new-energy-monitoring

# 安装 Git Hooks

```

安装完成后，每次代码提交和推送都会自动执行质量检查。

### 1.2 配置 IDE

#### 1.2.1 VS Code 配置

项目已包含 VS Code 配置文件 `.vscode/settings.json`，打开项目时会自动应用以下设置：

- 保存时自动格式化代码
- 保存时自动修复 ESLint 问题
- 保存时运行测试
- 保存时检查代码覆盖率

#### 1.2.2 GoLand/IDEA 配置

1. 打开 Settings (Preferences)
2. 导航到 Tools -> File Watchers
3. 添加以下 Watchers:
   - `go fmt`: 自动格式化 Go 代码
   - `goimports`: 自动导入包
   - `golangci-lint`: 代码检查

## 2. 质量门禁详解

### 2.1 第一层：代码提交门禁

#### 2.1.1 触发时机

每次执行 `git commit` 时自动触发。

#### 2.1.2 检查内容

| 检查项 | 工具 | 失败处理 | 执行时间 |
|--------|------|----------|----------|
| 敏感信息检查 | grep | 阻止提交 | < 1秒 |
| 代码格式化 | gofmt/ESLint | 阻止提交 | < 5秒 |
| 静态分析 | go vet | 阻止提交 | < 10秒 |
| 代码规范 | golangci-lint | 阻止提交 | < 30秒 |
| 安全漏洞 | gosec | 阻止提交 | < 20秒 |
| 快速测试 | go test -short | 阻止提交 | < 30秒 |
| 提交信息格式 | 正则表达式 | 阻止提交 | < 1秒 |

#### 2.1.3 提交信息规范

```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型 (type)**:
- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `style`: 代码格式调整
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建/工具变动
- `perf`: 性能优化
- `ci`: CI 配置变动
- `build`: 构建系统变动
- `revert`: 回退提交

**示例**:
```
feat(user): 添加用户登录功能

- 实现用户名密码登录
- 实现 Token 认证
- 添加登录日志记录

Closes #123
```

#### 2.1.4 跳过检查（不推荐）

```bash
# 跳过 pre-commit 检查
git commit --no-verify -m "feat: 紧急修复"

# 仅在紧急情况下使用！
```

### 2.2 第二层：PR 合并门禁

#### 2.2.1 触发时机

创建或更新 Pull Request 时自动触发。

#### 2.2.2 检查内容

| 检查项 | 工具 | 失败处理 | 执行时间 |
|--------|------|----------|----------|
| 代码审查 | 人工审查 | 阻止合并 | - |
| 单元测试 | go test | 阻止合并 | < 5分钟 |
| 集成测试 | go test | 阻止合并 | < 10分钟 |
| 测试覆盖率 | Codecov | 阻止合并 | < 1分钟 |
| 代码质量 | SonarQube | 阻止合并 | < 5分钟 |
| 安全扫描 | Trivy/Gosec | 阻止合并 | < 5分钟 |

#### 2.2.3 PR 创建流程

1. **创建分支**
   ```bash
   git checkout -b feature/new-feature
   ```

2. **编写代码和测试**
   - 遵循编码规范
   - 编写单元测试
   - 确保测试覆盖率达标

3. **提交代码**
   ```bash
   git add .
   git commit -m "feat(module): 添加新功能"
   git push origin feature/new-feature
   ```

4. **创建 Pull Request**
   - 在 GitHub/GitLab 上创建 PR
   - 填写 PR 模板
   - 关联相关 Issue

5. **等待 CI 检查**
   - 查看 CI 检查结果
   - 修复失败的检查项

6. **代码审查**
   - 等待团队成员审查
   - 响应审查意见
   - 修改代码并推送

7. **合并代码**
   - 所有检查通过
   - 至少 1 人 Approve
   - Squash and merge

#### 2.2.4 PR 模板填写

```markdown
## 变更说明
简要描述本次变更的内容

## 变更类型
- [x] 新功能 (feature)
- [ ] Bug修复 (bugfix)
- [ ] 重构 (refactor)
- [ ] 文档更新 (docs)
- [ ] 性能优化 (performance)
- [ ] 测试相关 (test)

## 测试情况
- [x] 已添加单元测试
- [x] 已添加集成测试
- [x] 已手动测试
- [x] 测试覆盖率达标

## 检查清单
- [x] 代码符合规范
- [x] 无 Lint 错误
- [x] 无安全漏洞
- [x] 已更新文档
- [x] 已更新 CHANGELOG

## 相关 Issue
Closes #123

## 截图/演示
（如有必要，提供截图或演示）
```

## 4. 常见问题处理

### 4.1 代码提交失败

#### 问题 1: 代码格式不符合规范

```bash
# 错误信息
错误: 以下文件格式不符合规范:
internal/service/user_service.go

# 解决方法
gofmt -w internal/service/user_service.go
```

#### 问题 2: Lint 检查失败

```bash
# 错误信息
错误: golangci-lint 检查失败

# 解决方法
# 1. 查看详细错误
golangci-lint run

# 2. 自动修复（部分问题）
golangci-lint run --fix

# 3. 手动修复剩余问题
```

#### 问题 3: 测试失败

```bash
# 错误信息
错误: 单元测试失败

# 解决方法
# 1. 运行测试查看详细错误
go test -v ./...

# 2. 修复测试失败的原因
# 3. 重新提交
```

### 4.2 PR 合并失败

#### 问题 1: 代码审查未通过

```bash
# 解决方法
# 1. 查看审查意见
# 2. 修改代码
# 3. 推送新的提交
git add .
git commit -m "fix: 根据审查意见修改代码"
git push origin feature/new-feature
```

#### 问题 2: 测试覆盖率不足

```bash
# 错误信息
错误: 覆盖率低于 70% 阈值

# 解决方法
# 1. 查看覆盖率报告
go test -coverprofile=coverage.out ./...
go tool cover -html=coverage.out

# 2. 添加缺失的测试用例
# 3. 重新推送
```

### 4.3 发布失败

#### 问题 1: 金丝雀验证失败

```bash
# 错误信息
金丝雀验证失败，自动回滚

# 解决方法
# 1. 查看监控指标，定位问题
```

## 5. 最佳实践

### 5.1 代码编写

1. **遵循编码规范**
   - 使用有意义的命名
   - 保持函数简短（< 50 行）
   - 避免重复代码

2. **编写高质量测试**
   - 测试覆盖所有边界条件
   - 使用 Mock 隔离依赖
   - 保持测试独立性

3. **及时提交**
   - 小步提交，频繁推送
   - 每次提交保持原子性
   - 写清楚提交信息

### 5.2 代码审查

1. **审查重点**
   - 功能正确性
   - 代码可读性
   - 性能影响
   - 安全性

2. **审查态度**
   - 对事不对人
   - 提出建设性意见
   - 及时响应审查请求

3. **审查效率**
   - 控制审查范围（< 400 行）
   - 使用工具辅助审查
   - 及时完成审查

### 5.3 测试执行

1. **本地测试**
   - 提交前运行快速测试
   - 推送前运行完整测试
   - 定期运行性能测试

2. **测试数据**
   - 使用独立的测试数据
   - 避免污染生产数据
   - 及时清理测试数据

3. **测试报告**
   - 关注测试覆盖率
   - 分析失败原因
   - 持续改进测试

### 5.4 发布管理

1. **发布计划**
   - 提前规划发布内容
   - 准备回滚方案
   - 通知相关人员

2. **发布执行**
   - 选择低峰时段发布
   - 监控关键指标
   - 及时响应告警

3. **发布后验证**
   - 验证核心功能
   - 收集用户反馈
   - 记录发布经验
