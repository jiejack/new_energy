# Caveman 模式使用指南

## 概述

Caveman 模式是一种高效通信模式，通过压缩响应内容减少 token 消耗，同时保持完整的技术准确性。

## 启用方式

### 手动触发

```
/caveman          # 启用默认级别(full)
/caveman lite     # 启用lite级别
/caveman ultra    # 启用ultra级别
```

### 自动触发

当满足以下条件时自动启用：
- Token 使用超过 50,000
- 对话轮次超过 20 轮

## 强度级别

| 级别 | 响应行数 | 代码行数 | 说明 | 适用场景 |
|------|----------|----------|------|----------|
| lite | ≤10 | ≤50 | 轻度压缩 | 一般讨论 |
| full | ≤5 | ≤30 | 中度压缩 | 代码开发 |
| ultra | ≤3 | ≤15 | 极度压缩 | 紧急修复 |

## 使用示例

### 标准模式 vs Caveman 模式

**标准模式响应**:
```
我已经分析了您的问题。根据错误日志显示，问题出现在数据库连接部分。
具体来说，连接池配置可能需要调整。建议您检查以下配置项：
1. 最大连接数
2. 空闲连接数
3. 连接超时时间

以下是修改建议的代码：
```

**Caveman 模式响应**:
```
DB连接问题。检查连接池配置：

```go
db.SetMaxOpenConns(100)
db.SetMaxIdleConns(10)
```
```

## 退出方式

```
/normal          # 返回标准模式
/standard        # 返回标准模式
```

## 注意事项

1. **信息完整性**: Caveman 模式不会丢失关键技术信息
2. **代码质量**: 代码本身不会被压缩，只是减少解释文字
3. **紧急情况**: ultra 级别适用于紧急修复，日常开发建议使用 full 或 lite

## 配置文件

配置文件位置: `.trae/caveman-config.json`

可根据项目需求自定义：
- 触发条件
- 响应行数限制
- 缩写规则
- 退出命令

## 效果评估

| 指标 | 标准模式 | Caveman模式 | 节省 |
|------|----------|-------------|------|
| Token消耗 | 100% | 25-40% | 60-75% |
| 响应时间 | 100% | 60-70% | 30-40% |
| 信息完整性 | 100% | 100% | 0% |
